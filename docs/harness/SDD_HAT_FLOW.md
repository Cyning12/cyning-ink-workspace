# SDD 与帽子编号流程（简版）

> **用途**：把「10 / 20 / 22 / 30 / 40 / 50」串成可执行的协作顺序；说明 **22 签收** 与 **50 复检** 的边界、**40 自检** 与 **22 审核** 的分工，以及常见 **打回** 回流。  
> **真值**：[`HARNESS_V2_PLAN.md`](HARNESS_V2_PLAN.md)（第三节角色帽子、第五节 task 字段）、[`prompts/README.md`](prompts/README.md)（帽子文件与调用模板）、[`reviews/README.md`](reviews/README.md)（签收与 task 状态对齐）。

---

## 1. 编号与职责（一览）

| 编号 | 帽子 | 产出 / 硬约束（摘要） |
|------|------|------------------------|
| **10** | 需求 / 任务分析 | 写或改 task：验收可观测、`failure_paths`、非范围、依赖链接；承接 22 的回填清单 |
| **20** | 规格 / 任务短评 | **对话内**缺口与可测性短评；**不**强制落 `reviews/`；书面签收走 **22** |
| **22** | 任务审核 | **每次**在 [`reviews/`](reviews/README.md) 落盘；**R1 / R2 / …**；**「签收 / 关闭」节 = 任务正式终点点**（与 task 状态一致） |
| **30** | 执行编码 | 按 task 必读、`test_strategy`；缺项可 **拒开工**（仅阻塞清单） |
| **40** | 自检（执行者） | 跑 task 所列验证命令；**回填** task 内 **`### 自检结论（执行者）`** |
| **50** | 独立复检 / 全局验收 | 以 **diff + 日志 + 验收表**（及可选 `freeze_id`、人签 checklist）为主；**换上下文**抽查，**不替代** CI |

---

## 2. 「22 通过」之后是不是一定是「50」？

**不是。**

- **`22` 终轮**写出 **`reviews/`** 的 **「签收 / 关闭」** 时，在 Harness 口径下 **任务审核链已可收口**（见 [`prompts/README.md`](prompts/README.md) 使用方式第 4 步）。
- **`50`** 为 **可选加强**：高敏变更、需要独立视角对照证据、或要对齐 **`freeze_id` + 人签 checklist** 时再开；**不是** `22` 之后的法定下一步。

---

## 3. 「40 自检」与「22 审核」怎么分工？

- **`40`**：把「声称完成」变成 **可核对命令与验收摘要**，并写入 task **自检结论**；**不**承担书面 **签收 / 关闭**。
- **`22`（如 R2）**：对照 diff、日志、自检结论与 **task 验收条**，在 **`reviews/`** 落盘并给出 **签收 / 关闭** 或 **阻塞清单**。

因此常见顺序是：**30 → 40 → 22（收口审）**；`22` 前是否再插一轮 `22(R1)` 视任务是否已预审而定。

---

## 4. 与 SDD 一句话对齐

规划中的 SDD 摘要（Inform / Constrain / Verify 之上）：**SPEC → task → 短评 → 任务审核落盘 → 执行 → 自检 →（可选）复检 → 人检 checklist**。

映射到编号时，短评多为 **20**，任务审核为 **22**，执行为 **30**，自检为 **40**，独立复检 / 全局验收为 **50**；**10** 承担 task 侧塑形与回填。

---

## 5. 推荐路径（用数字表示）

以下 **`22(Rn)`** 表示第 n 轮审核文档（R1、R2…），**`[50?]`** 表示按团队策略可选。

### 5.1 较完整（含预审与收口审）

```text
10 → 20 → 22(R1) → 30 → 40 → 22(R2 签收) → [50?]
```

示例（与对话中举例一致）：`10 → 20 → 22 → 30 → 40 → 22 → 50` 表示在 **R2 签收后** 仍加做 **50**，属于 **合法变体**，非唯一标准形。

### 5.2 极简（小改动、风险可控）

```text
10 → 22(R1) → 30 → 40 → 22(R2 签收)
```

可省略 **20**（短评在对话完成即可）；**50** 也可省略。

### 5.3 打回一次：审核阶段不通过（改 task / SPEC）

```text
10 → 20 → 22(R1 阻塞) → 10（改 task）→ 22(R1' 或 R2 视团队命名) → 30 → 40 → 22(终轮签收) → [50?]
```

若审查要求改 **冻结点 / 契约**，仍建议经 **10** 或 SPEC 变更说明后再 **22** 出 **R+1** 文档（见 [`prompts/README.md`](prompts/README.md) 第 4 步）。

### 5.4 打回一次：实现或自检失败（少改 task 正文）

```text
10 → [20?] → 22(R1) → 30 → 40(失败) → 30（修）→ 40 → 22(R2 签收) → [50?]
```

若 **R1 已放行** 但实现方向错误，也可能回流 **`10`** 收敛范围后再 **`22'`**。
