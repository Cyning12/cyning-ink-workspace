# Harness 通则：流程关闭时的执行路线与 Commit 回溯

> **适用范围**：本轮判定 **无下一棒**、Harness 子流程 **可关闭** 时（任务签收、阶段收口、独立复检建议合并且不再返工等）。  
> **与 [`HANDOFF_AUTO_COMMIT.md`](HANDOFF_AUTO_COMMIT.md) 关系**：有下一棒 → 产出 Prompt + 本轮 commit；**无下一棒** → **不**强行编造下一棒 Prompt，改产出本章 **执行路线 + commit 表**（本轮工件仍须按 AUTO_COMMIT 提交，若确有落盘）。  
> **真值层级**：Guides；审查终轮、task `done`、合并决策以各帽规则为准。

---

## 1. 何时视为「可关闭、无下一棒」

满足 **其一** 即可在本轮收尾采用本章（而非「下一棒可复制 Prompt」）：

| 场景 | 典型判定者 | 说明 |
|------|------------|------|
| **任务审核终轮签收** | `22` | 审查 md **「签收 / 关闭」** 写明 task 可结束；无 R+1、无回填清单 |
| **task 工程阶段收口** | `22` / `40` / `50` | task 内阶段（如 P2-0）已自检 pass，且元信息/§10 标明 **下一棒角色 = 无** 或仅维护性跟进 |
| **独立复检 / 全局验收通过** | `50` | 建议合并且无阻塞；不再打回执行帽 |
| **用户明示** | 任意帽 | 「本 task 结束」「不要下一棒 Prompt」 |

**仍须下一棒时**：继续用各 `TEMPLATE-*-invoke` §3 + `HANDOFF_AUTO_COMMIT`，**不得**用空回溯表代替 Prompt。

---

## 2. 必须输出什么

### 2.1 对话（必有）

在回复末尾设独立小节 **「执行路线与 Commit 回溯」**（标题勿改，便于检索）。内容 **至少** 包含：

1. **一句结论**：流程为何关闭（签收 / 阶段完成 / 复检通过等）。  
2. **执行路线表**（按时间顺序）：见 §2.2。  
3. **分仓 commit 索引**：见 §2.3。  
4. **关联工件路径**：task、终轮 `reviews/*_audit_*`、关键 `invokes/invoke_*`（相对 `Projects/` 或注明子仓根）。

### 2.2 执行路线表（建议列）

| 序号 | 阶段 / 帽子 | 关键动作 | 落盘工件（相对路径） | 对应 commit |
|------|-------------|----------|----------------------|-------------|
| 1 | `10` 需求 | … | `…/task_*.md` | `子仓@abc1234` 或 `—` |
| 2 | `22` R1 | … | `…/reviews/task_*_audit_R1_*.md` | … |
| … | … | … | … | … |

- **阶段**：可与 task 内 P2-x、Harness 轮次 R1/R2 对齐。  
- **对应 commit**：写 **`仓库简称 @ 7 位 short-hash`**（如 `api-python@3214908`）；同阶段多 commit 用逗号分隔；仅对话无提交填 **`—`**。  
- **禁止**只写「见 git 历史」而不列表。

### 2.3 分仓 Commit 索引（建议格式）

按 **git 根** 分组，**从新到旧** 列出与本 task / `freeze_id` / slug **相关** 的提交（**不必**罗列无关 WIP）：

```text
### api-python（ai-ink-brain-api-python）
- 135c655 feat(tech_graph): P2-1 CI 与导出/等价测试对齐 graph_v2
- 3214908 docs(tech_graph): P2-0 自检签收；导出器切 graph_v2 构建
- 9d9161d feat(tech_graph): P2-0 graph_v2 schema 与等价检查草案
…

### Projects（工作区根，若适用）
- 4a60394 docs(harness): 下一棒 Prompt 后自动 commit 通则
…
```

**采集方式（Agent）**：

```bash
# 子仓内按时间范围或关键词（示例）
git -C ai-ink-brain-api-python log --oneline -20 -- docs/tasks/active/task_<slug>.md docs/harness/
```

若环境无法执行 git：根据 task §10、invoke/review 元信息 **手工汇总** 已知 hash，并标注 **「待用户本地 `git log` 补全」**。

### 2.4 落盘（建议 · 终轮审查 / task）

| 落点 | 要求 |
|------|------|
| **终轮** `reviews/task_*_audit_R*_*.md` | 在 **「签收 / 关闭」** 之后增 **「执行路线与 Commit 回溯」** 小节（表 + 分仓索引，与对话 **语义一致**） |
| **task** `§10` 或 **「审查与阶段状态」** | 增一行 **「关闭回溯」** 链向终轮审查该小节，或粘贴 **缩短版** 表（至少含最后 3 个阶段 + 末条 commit） |
| **无审查驱动的关闭**（如仅 `50`） | 可只在对话输出；若落盘复检报告 md，则写入报告末节 |

**无下一棒时**：审查 md **可不**含 **「下一棒可复制 Prompt」** 小节；**禁止**用「无」占位糊弄，应 **省略该节** 并 **必有** 本章回溯节。

## 3. 与各帽子的分工

| 帽子 | 关闭回溯谁写 |
|------|----------------|
| **22** 终轮签收 | **主责**：审查 md + 对话双写 |
| **50** 建议合并且无返工 | **主责**：对话；可选 PR 评论粘贴 |
| **40** 最后一阶段自检后 task 标明结束 | **共责**：回填 task §10 缩短版 + 建议触发 `22` 终轮 |
| **30** | 一般不单独关闭整 task；阶段完成 → 交 `40` / `22` |
| **10 / 20** | 不关闭整 task；若用户仅取消草案，输出 **阻塞清单** 即可，无需 commit 表 |

---

## 4. 执行顺序（关闭轮）

1. 完成本轮落盘（审查签收节 / task 状态 / 自检表等）。  
2. 按 [`HANDOFF_AUTO_COMMIT.md`](HANDOFF_AUTO_COMMIT.md) 提交 **本轮** 路径（含回溯表写入的 review/task）。  
3. 汇总 **自 task 创建或上轮回溯以来** 的相关 commit，填 §2.2–2.3。  
4. 对话输出 **「执行路线与 Commit 回溯」**；**禁止**再附下一棒 Prompt。  
5. 建议用户将 task 头部 `status` 置为 `done`（若尚未）；**不**代用户 push。
